# Lessons

- No project-specific corrections recorded yet.
- When a side panel mixes summary data and creation controls, visually separate day-level information from block-level inputs with explicit section titles and container boundaries before adding more fields.
- Weekly overview cells should not become disconnected from assigned block data; show compact block references in the relevant day/category cells and let coaches expand them for notes.
- If day-level summary fields are surfaced in the inspector, assume the coach may need to edit them there and sync those edits back to the board header.
- Keep creation forms collapsed when the primary sidebar job is fast reference; expose a compact action and open the heavier form only on request.
- In grid cells that contain secondary reference cards, make the primary editable label compact so the reference content stays visually connected to the label.
- Dashboard summary metrics should be sized to their decision value; compact status pills are better than large cards when the board itself is the primary work surface.
- Remove redundant subheadings when the main page title already establishes context; keep accessibility labels without spending visible space.
- Do not let assigned workload blocks inherit neutral colors from static row labels; block-bearing cells need color derived from planned AU or an explicit workload setting.
- Keyword-driven load guidance must remain secondary once a coach enters planned AU; coach-entered AU is the primary source of truth for day-load evaluation.
- If a workload class is present in the DOM, make sure component-specific CSS selectors do not mask it with a stronger default background.
- For large production migrations, split frontend and backend into concurrent worker-owned scopes early, then reserve the main thread for integration and verification.
